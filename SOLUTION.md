# Решение

Сервис анализа нарушений ПДД. Читает проезды транспортных средств из брокера сообщений, анализирует треки и фиксирует проезд на красный свет в базе данных.

## Запуск
```bash
go run cmd/app/main.go
```

Сервис завершится автоматически после обработки 200 сообщений или по Ctrl+C.

## Архитектура

Гексагональная архитектура (ports & adapters) — сервис работает только с интерфейсами, не знает про Cat и Dog напрямую. Замена на реальный брокер или БД потребует только нового адаптера.
```
cmd/app           — точка входа, инициализация, graceful shutdown
internal/config   — конфигурация с дефолтными значениями
internal/ports    — интерфейсы Broker и Storage
internal/adapters — реализации для Cat и Dog
internal/retry    — утилита повторных попыток
internal/service  — бизнес-логика, обработка сообщений
```

## Ключевые решения

**Параллельная обработка** — Cat и Dog симулируют задержки 0.5–2 сек. Запускаем пул из 4 воркеров чтобы не упираться в скорость одного потока.

**Retry** — Dog с вероятностью 1/10 возвращает `ErrInternal`. Повторяем вставку до 3 раз с паузой 100мс. На `ErrHasNoConn` ретраев нет — это не временная ошибка.

**Поиск последней точки** — трек намеренно перемешан. Один проход O(n) вместо сортировки.

**Graceful shutdown** — сервис завершается по сигналу ОС или при достижении лимита сообщений. Все горутины дожидаются завершения текущей обработки перед выходом.

**LogLevel** — на `debug` логируются все проезды, на `info` только нарушения.
